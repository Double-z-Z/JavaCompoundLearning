# 代码修改规范

## 修改前必须回答的问题

在进行任何代码修改前，必须能够清晰回答以下问题：

### 1. 为什么要修改？
- **需求因素**：这个修改是为了满足什么需求？
- **问题因素**：当前代码存在什么问题？
- **设计因素**：是否符合整体架构设计？

### 2. 修改的影响范围？
- **直接影响**：修改了哪些类/方法？
- **间接影响**：会影响哪些其他模块？
- **接口契约**：是否改变了公共API的语义？

### 3. 修改的合理性？
- **设计一致性**：是否符合项目的设计原则？
- **语义正确性**：是否保持了方法/类的原始语义？
- **可维护性**：是否增加了代码的复杂度？

## 禁止的修改行为

### ❌ 禁止：为了局部逻辑修改全局定义
- **例子**：为了调整A地方的逻辑，修改B方法的全局定义
- **原因**：破坏封装，影响其他调用方
- **正确做法**：在A地方做适配，或添加新方法来满足需求

### ❌ 禁止：改变方法/类的原始语义
- **例子**：将`shutdownNow()`（强制关闭）改为优雅关闭
- **原因**：破坏接口契约，调用方期望的行为被改变
- **正确做法**：添加新方法或调整调用逻辑，不改变原方法语义

### ❌ 禁止：暴露内部实现细节
- **例子**：让Worker直接获取其他Worker的引用
- **原因**：破坏封装，增加耦合
- **正确做法**：通过Boss提供需要的功能，不暴露内部结构

### ❌ 禁止：未经充分验证的修改
- **例子**：修改后没有运行测试验证
- **原因**：可能引入回归问题
- **正确做法**：修改后必须运行相关测试

## 修改注释规范

任何修改都必须添加注释，说明：

```java
// 修改原因：[需求/问题/设计因素]
// 影响范围：[直接/间接影响]
// 验证方式：[如何验证修改的正确性]
// 注意事项：[其他需要注意的事项]
```

## 修改流程

1. **分析阶段**
   - 理解当前代码的设计意图
   - 识别问题的根本原因
   - 评估不同修改方案的影响

2. **设计阶段**
   - 选择影响范围最小的方案
   - 确保不破坏接口契约
   - 考虑可维护性和扩展性

3. **实施阶段**
   - 添加清晰的修改注释
   - 保持代码风格一致
   - 最小化修改范围

4. **验证阶段**
   - 运行相关测试
   - 检查是否有回归问题
   - 验证修改是否解决了原始问题

## 常见错误模式

### 错误1：混淆"直接原因"和"根本原因"
- **直接原因**：测试失败了
- **根本原因**：消息编码错误导致客户端无法解析
- **正确做法**：修复编码问题，而不是调整测试期望

### 错误2：破坏封装层次
- **错误**：Worker直接访问其他Worker的内部状态
- **正确**：通过Boss协调，Worker只关注自己的职责

### 错误3：改变语义来适应调用方
- **错误**：将`shutdownNow()`改为优雅关闭
- **正确**：调整调用逻辑，让`shutdownNow()`在正确时机被调用

### 错误4：过度设计
- **错误**：为了一个小问题引入复杂的抽象
- **正确**：简单直接的解决方案，必要时再重构
