# 双重检查模式

## 一句话定义
在**用户态和内核态各做一次检查**，防止竞态条件导致线程错误阻塞的设计模式。


## 核心机制
1. **第一次检查（用户态）**：用CAS尝试获取锁，成功则直接返回
2. **第二次检查（内核态）**：进入内核后再次检查状态，避免"检查后被修改"的竞态


## 为什么重要
没有双重检查会导致：
```
线程A: 检查*futex_var == 0
    ↓
线程B: futex_var = 1（加锁）
    ↓
线程A: CAS失败，进入futex_wait()
    ↓
如果不二次检查 → 线程A阻塞（但锁已可用！）→ 死锁
二次检查后 → 发现值已变，直接返回重新尝试
```


## 与已学知识的关联
- [[futex]] → 双重检查是futex的核心实现机制
- CAS → 第一次检查依赖CAS的原子性
- 竞态条件 → 双重检查解决的经典问题


## 常见误区
- ❌ "第二次检查是多余的" → 实际上防止了严重的竞态条件
- ❌ "只在锁机制中使用" → 实际上广泛应用于单例模式等场景


## 代码示例
```c
// 用户态
if (atomic_compare_and_swap(&futex_var, 0, 1) == 0) {
    return 0;  // 快速路径
}

// 内核态
SYSCALL_DEFINE2(futex_wait, u32 *uaddr, u32 val) {
    if (*uaddr != val) {  // 第二次检查！
        return -EWOULDBLOCK;  // 值已变化，无需阻塞
    }
    // 真正需要阻塞...
}
```
