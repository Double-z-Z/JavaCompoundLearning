# 线程阻塞

## 一句话定义
线程在等待某个条件满足时，**主动让出CPU时间片**，从RUNNABLE状态转为WAITING/TIMED_WAITING状态的过程。


## 核心机制
1. **分层调用**：Java(park) → JVM(Parker) → glibc(pthread) → 内核(futex)
2. **用户态优先**：无竞争时用CAS，有竞争时才进入内核态阻塞
3. **状态分离**：唤醒（WAITING→RUNNABLE）≠ 调度（分配CPU）


## 完整调用链路
```
LockSupport.park()          // Java层
    ↓ JNI
JVM_Unsafe_Park()           // JVM层
    ↓arker::park()              // C++层
    ↓
pthread_cond_wait()         // glibc层
    ↓ syscall
futex(FUTEX_WAIT)           // 内核层
    ↓
schedule() → context_switch() // 调度器
```


## 为什么重要
- 理解`synchronized`、`ReentrantLock`的底层实现
- 阻塞有开销（~1-5微秒），能避免就避免
- 调试并发问题（死锁、性能瓶颈）的基础


## 与已学知识的关联
- [[futex]] → 线程阻塞在内核层的实现
- 线程状态 → 阻塞涉及的状态转换
- [[Parker]] → JVM层的阻塞封装
- [[用户态与内核态切换]] → 阻塞的代价来源


## 常见误区
- ❌ "park()直接调用futex" → 中间还有pthread库封装
- ❌ "阻塞=立即失去CPU" → 状态变更和调度是分开的
- ❌ "唤醒=立即执行" → 唤醒后还要等待调度器选择


## 代码示例
```java
// 线程阻塞的典型场景
synchronized(lock) {
    while (conditionNotMet) {
        lock.wait();  // 阻塞，释放锁
    }
}

// 对应的唤醒
synchronized(lock) {
    conditionMet = true;
    lock.notify();  // 唤醒等待线程
}
```
