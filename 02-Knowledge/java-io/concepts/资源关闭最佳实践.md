# 资源关闭最佳实践

## 一句话定义
资源关闭是释放系统资源的关键操作，需要根据场景选择适当的异常处理策略。


## 核心理解

### 资源关闭的两种场景

| 场景 | 特点 | 异常处理策略 |
|------|------|-------------|
| **业务逻辑** | 关闭是业务流程的一部分 | 必须处理异常，可能需要重试或回滚 |
| **资源清理** | 在finally块中清理资源 | 可以忽略异常，不影响主流程 |

### 为什么close()要声明throws IOException？

```java
// close()声明throws的原因
public void close() throws IOException;
```

**原因**：
1. **契约完整性**：close()可能涉及IO操作（如刷新缓冲区到磁盘）
2. **调用者知情权**：调用者需要知道可能发生的异常
3. **不同场景不同处理**：业务逻辑需要处理，资源清理可以忽略

### 生产环境规范

```java
/**
 * 安静关闭 - 用于资源清理场景
 * 忽略异常，不干扰主流程
 */
public static void closeQuietly(Closeable closeable) {
    if (closeable != null) {
        try {
            closeable.close();
        } catch (IOException ignored) {
            // 资源清理场景，忽略异常
        }
    }
}

/**
 * 带日志的关闭 - 用于需要监控的场景
 */
public static void closeWithLogging(Closeable closeable, Logger logger) {
    if (closeable != null) {
        try {
            closeable.close();
        } catch (IOException e) {
            logger.warn("关闭资源失败: {}", closeable, e);
        }
    }
}
```

## 我的误区与疑问

### ❌ 误区：所有close()异常都必须处理
**纠正**：资源清理场景可以忽略异常

### ❌ 误区：closeQuietly()是"偷懒"
**纠正**：这是有意识的设计选择，区分业务逻辑和资源清理


## 代码与实践

### 工具类实现
```java
public final class IoUtils {
    private IoUtils() {
        throw new AssertionError("工具类禁止实例化");
    }
    
    /**
     * 安静关闭单个资源
     */
    public static void closeQuietly(Closeable closeable) {
        if (closeable != null) {
            try {
                closeable.close();
            } catch (IOException ignored) {
                // 资源清理场景，忽略异常
            }
        }
    }
    
    /**
     * 安静关闭多个资源
     */
    public static void closeQuietly(Closeable... closeables) {
        if (closeables != null) {
            for (Closeable closeable : closeables) {
                closeQuietly(closeable);
            }
        }
    }
}
```

## 深入思考

💡 **设计哲学**：
- close()声明throws是"告知义务"
- closeQuietly()是"场景选择"
- 两者不矛盾，是不同抽象层次的设计

💡 **类比**：
> 就像垃圾回收：系统有义务提供回收机制，但你可以选择"自动回收"或"手动处理"。
