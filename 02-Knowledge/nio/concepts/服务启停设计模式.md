# 服务启停设计模式

## 一句话定义
通过状态管理和分层关闭策略，实现服务的可控启动和优雅停止，确保资源正确释放且不影响用户体验。


## 核心理解

### 四大核心要求

| 要求 | 含义 | 实现方式 |
|------|------|---------|
| **幂等性** | 多次调用不会出问题 | 状态检查 + 同步 |
| **状态管理** | 明确的生命周期 | 枚举状态 + 状态流转校验 |
| **优雅关闭** | 先停止接收新请求，再等待现有请求完成 | shutdown() → awaitTermination() |
| **资源清理** | 确保所有资源被正确释放 | finally块 + closeQuietly模式 |

### 状态流转设计

```
NEW → STARTING → RUNNING → STOPPING → STOPPED
 │       │          │          │         │
 │       │          │          │         └─ 最终状态
 │       │          │          └─ 正在关闭资源
 │       │          └─ 可接受连接/请求
 │       └─ 初始化中
 └─ 刚创建
```

### 优雅关闭 vs 立即关闭

| 类型 | 命名惯例 | 适用场景 | 实现复杂度 |
|------|---------|---------|-----------|
| **优雅关闭** | `shutdown()` / `stop()` | 正常维护、版本升级 | 中等 |
| **立即关闭** | `shutdownNow()` / `terminate()` | 严重漏洞、系统故障 | 较高 |

这让我想起 [[线程池]] 的设计 —— Java 的 `ExecutorService` 正是这样设计的：
- `shutdown()`: 停止接收新任务，等待现有任务完成
- `shutdownNow()`: 尝试中断所有正在执行的任务

## 我的误区与疑问

### ❌ 误区：每个服务都必须提供 shutdown() 和 shutdownNow() 两个接口
**纠正**：是否需要双接口取决于服务的关键程度：
- 核心基础设施（连接池、消息队列）：需要双接口
- 普通业务服务：通常只提供优雅关闭
- 测试/临时项目：简单关闭即可，直接 kill 进程也够用

### ❌ 误区：shutdownNow() 是"更好的"关闭方式
**纠正**：立即关闭是应急手段，不是常规操作。它可能导致：
- 数据不一致
- 请求中断
- 资源泄漏


## 代码与实践

### ChatServer 中的实现

```java
public enum State {
    NEW, STARTING, RUNNING, STOPPING, STOPPED
}

// 幂等性保证
public synchronized void start() throws IOException, TimeoutException {
    if (state != State.NEW && state != State.STOPPED) {
        throw new IllegalStateException("服务器已经启动");
    }
    state = State.STARTING;
    // ... 初始化
    state = State.RUNNING;
}

// 优雅关闭
public synchronized void stop() {
    if (state != State.RUNNING) return;
    state = State.STOPPING;
    shutdown();
    state = State.STOPPED;
}

private void shutdown() {
    // 先业务资源，后线程池
    if (clientManager != null) clientManager.shutdown();
    if (serverExecutor != null) {
        serverExecutor.shutdown();
        try {
            if (!serverExecutor.awaitTermination(5, TimeUnit.SECONDS)) {
                serverExecutor.shutdownNow();
            }
        } catch (InterruptedException e) {
            serverExecutor.shutdownNow();
        }
    }
}
```

### 带超时的优雅关闭（进阶）

```java
/**
 * 带超时的优雅关闭
 * @return true=优雅关闭成功, false=超时转为强制关闭
 */
public boolean shutdown(long timeout, TimeUnit unit) {
    shutdown(); 
    try {
        if (!awaitTermination(timeout, unit)) {
            logger.warn("优雅关闭超时，转为强制关闭");
            shutdownNow();
            return false;
        }
        return true;
    } catch (InterruptedException e) {
        shutdownNow();
        return false;
    }
}
```


## 深入思考

💡 如果服务发现自身存在安全漏洞，关闭策略应该如何选择？
- 数据泄露类漏洞：先拒绝新请求，让现有请求完成
- 远程代码执行漏洞：立即停止一切

💡 JVM ShutdownHook 和显式调用 stop() 的关系？
- ShutdownHook 捕获的是 OS 信号（kill/Ctrl+C）
- 它执行的逻辑应该和显式 stop() 一致
- 但 OS 最终可能强制终止，不管是否执行完
