# 测试设计

> NIO聊天室项目的测试设计经验总结

---

## 测试分类策略

### 1. 单元测试（Unit Tests）
- **特点**：快速、无IO、独立
- **范围**：单个类/方法
- **执行时间**：< 100ms
- **示例**：`ChatMessageEncoderTest`、`ChatMessageDecoderTest`

### 2. 集成测试（Integration Tests）
- **特点**：需要服务器运行、涉及网络IO
- **范围**：多组件协作
- **执行时间**：秒级
- **示例**：`BroadcastTest`、`GracefulShutdownTest`

### 3. 性能测试（Performance Tests）
- **特点**：测量吞吐量、延迟、资源使用
- **范围**：系统整体性能
- **执行时间**：分钟级
- **示例**：`BroadcastPerformanceTest`

### 4. 端到端测试（E2E Tests）
- **特点**：完整业务流程
- **范围**：用户场景
- **执行时间**：分钟级

## NIO测试的特殊挑战

### 挑战1：端口冲突
**问题**：多个测试同时运行，端口被占用  
**解决**：
- 使用随机端口（`BASE_PORT + random * 1000`）
- 顺序执行（Maven Surefire配置）
- 独立JVM（forkMode=always）

### 挑战2：异步事件验证
**问题**：NIO事件异步触发，难以断言  
**解决**：
- 使用 `CountDownLatch` 等待事件
- 设置合理超时
- 轮询检查状态

### 挑战3：资源清理
**问题**：Selector、Channel未正确关闭，影响后续测试  
**解决**：
- `@After` 强制清理
- `awaitTermination()` 等待关闭完成
- 端口释放等待（sleep 50-100ms）

### 挑战4：竞态条件
**问题**：测试执行顺序影响结果  
**解决**：
- 每个测试独立端口
- 等待注册完成（sleep或同步机制）
- 避免共享状态

## 测试组织最佳实践

### 目录结构
```
src/test/java/com/example/
├── unit/           # 单元测试
├── integration/    # 集成测试
├── performance/    # 性能测试
└── e2e/           # 端到端测试
```

### Maven配置
```xml
<profile>
    <id>unit-tests</id>
    <activation><activeByDefault>true</activeByDefault></activation>
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-surefire-plugin</artifactId>
                <configuration>
                    <includes>
                        <include>**/unit/**/*Test.java</include>
                    </includes>
                </configuration>
            </plugin>
        </plugins>
    </build>
</profile>
```

### 测试命名规范
- 单元测试：`*Test.java`
- 集成测试：`*IntegrationTest.java`
- 性能测试：`*PerformanceTest.java`

## 测试冗余识别

### 识别标准
1. **功能重叠**：测试相同代码路径
2. **场景重复**：验证相同行为
3. **价值递减**：新增测试不增加信心

### 本次整理的冗余
| 测试类 | 冗余原因 | 处理方式 |

| `GracefulShutdownDebugTest` | 调试专用，非正式测试 | 删除 |
