# 线程池风格接口设计

## 一句话定义
模仿 Java `ExecutorService` 的接口设计模式，将"触发关闭"和"等待完成"分离，提供灵活且一致的关闭语义。


## 核心理解

### 线程池接口回顾

Java `ExecutorService` 提供三个核心关闭方法：

| 方法 | 语义 | 阻塞性 |
|------|------|--------|
| `shutdown()` | 触发优雅关闭，不再接受新任务，但完成已提交任务 | 非阻塞 |
| `shutdownNow()` | 立即关闭，尝试中断正在执行的任务 | 非阻塞 |
| `awaitTermination(timeout, unit)` | 等待关闭完成 | 阻塞 |

### 设计优势

1. **职责分离**：触发关闭 vs 等待完成，调用者可以自由组合
2. **灵活控制**：可以先触发关闭，做其他事情，再等待
3. **超时控制**：`awaitTermination` 支持超时，避免无限等待
4. **一致性**：与 Java 标准库保持一致，降低学习成本

### 应用到 NIO 服务

将 `ChatServer`、`ClientManager`、`ServerHandler` 的接口统一为线程池风格：

```java
// ChatServer
public void shutdown();                           // 触发关闭
public boolean awaitTermination(long timeout, TimeUnit unit);  // 等待完成
public boolean stop(long timeoutMs);              // 便捷方法：触发+等待

// ClientManager
public void shutdown();
public boolean awaitTermination(long timeout, TimeUnit unit);

// ServerHandler
public void shutdown();
public boolean awaitTermination(long timeout, TimeUnit unit);
```


## 关键关联

- [[ExecutorService-线程池]] - 关联原因：直接借鉴 Java 标准库的成熟设计模式
- [[状态机模式-服务生命周期]] - 关联原因：`shutdown()` 触发状态流转（RUNNING → SHUTTING_DOWN）
- [[异步编程-Future模式]] - 关联原因：`awaitTermination` 类似于 `Future.get(timeout, unit)` 的等待语义


## 我的误区与疑问

- ❌ **误区**：最初设计 `shutdown(timeoutMs)` 既触发又等待，导致无法灵活控制
- ✅ **纠正**：拆分为 `shutdown()` + `awaitTermination()`，与线程池保持一致


## 代码与实践

### 使用方式对比

**❌ 旧设计（不灵活）**
```java
// 只能同步等待
boolean graceful = server.shutdown(5000);
```

**✅ 新设计（灵活）**
```java
// 方式1：异步关闭
server.shutdown();
// ... 做其他事情 ...
boolean stopped = server.awaitTermination(5, TimeUnit.SECONDS);

// 方式2：同步关闭（便捷方法）
boolean stopped = server.stop(5000);
```

### 层级调用关系
```
ChatServer.stop()
    ├─ ChatServer.shutdown()
    │   ├─ ClientManager.shutdown()
    │   │   └─ ServerHandler.shutdown()  // 设置 SHUTTING_DOWN 状态
    │   └─ serverExecutor.shutdown()
    │
    └─ ChatServer.awaitTermination()
        ├─ ClientManager.awaitTermination()
        │   └─ ServerHandler.awaitTermination()  // 等待 stopFuture
        └─ serverExecutor.awaitTermination()
```
