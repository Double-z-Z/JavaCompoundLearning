# Redis Cluster 模式

## 一句话定义
Redis Cluster 是 Redis 的分布式解决方案，通过数据分片（Sharding）将数据分布到多个节点，实现水平扩展和高可用。


## 核心理解

### 数据分片（Sharding）

```
16384 个哈希槽（Hash Slot）均匀分布到多个 Master 节点：
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Master1 │  │ Master2 │  │ Master3 │
│ 0-5460  │  │ 5461-10922│ │10923-16383│
│         │  │         │  │         │
│ key1    │  │ key2    │  │ key3    │
│ key4    │  │ key5    │  │ key6    │
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └────────────┴────────────┘
            slot = CRC16(key) % 16384
```

**为什么是 16384？**
- 2^14 = 16384，用 14 位表示
- 心跳包中携带槽位图，16384/8=2KB
- 平衡了粒度和网络开销

### 架构拓扑

```
┌─────────────────────────────────────────┐
│           Redis Cluster 架构             │
│                                         │
│  ┌─────────┐      ┌─────────┐          │
│  │ MasterA │◄────►│ SlaveA1 │          │
│  │ (主)    │      │ (从)    │          │
│  │ 槽 0-5000│     └─────────┘          │
│  └────┬────┘      ┌─────────┐          │
│       │           │ SlaveA2 │          │
│       │           │ (从)    │          │
│       │           └─────────┘          │
│       │                                │
│       │      ┌─────────┐      ┌─────────┐
│       └──────┤ MasterB │◄────►│ SlaveB1 │
│              │ (主)    │      │ (从)    │
│              │槽5001-10000    └─────────┘
│              └────┬────┘               │
│                   │                    │
│                   │      ┌─────────┐   │
│                   └──────┤ MasterC │   │
│                          │ (主)    │   │
│                          │槽10001-16383 │
│                          └─────────┘   │
│                                         │
│  特点：                                  │
│  ├── 3 个 Master，数据分片存储            │
│  ├── 每个 Master 有 1-N 个 Slave         │

│  └── 无中心架构，节点间 Gossip 协议通信    │
│                                         │
└─────────────────────────────────────────┘
```

### Gossip 协议

```
无中心架构，节点间如何知道彼此状态？

Gossip 协议（流言协议）：
├── 每个节点随机选择其他节点发送信息
├── 信息包括：节点状态、槽位分布、故障信息
└── 最终一致性，像病毒传播一样扩散

消息类型：
├── PING：探测节点是否存活
├── PONG：回应 PING，携带自身状态（整合认知）
├── MEET：新节点加入集群
└── FAIL：标记节点故障

传播的信息内容：
├── 节点状态：ID、IP、端口、角色（Master/Slave）、状态（在线/下线）
├── 槽位映射：每个节点负责的槽位 Bitmap（16384 bit = 2KB）
├── 复制偏移量：Slave 的同步进度（用于故障转移选主）
├── Epoch 信息：configEpoch（配置版本）、currentEpoch（逻辑时钟）
└── 故障报告：PFAIL（疑似故障）、FAIL（确认故障）

冲突解决（反熵机制）：
├── 版本号优先：高 Epoch 覆盖低 Epoch
├── 时间戳辅助：Epoch 相同时，新的优先
├── 单调性保证：Epoch 只增不减，不会循环
└── 收敛性：有限状态空间 + 定期全量同步，确保最终一致

优势：
├── 去中心化，无单点故障
├── 可扩展，节点数增加不影响性能（O(log n) 轮传播）
└── 容错，部分网络分区不影响整体

反熵（Anti-Entropy）：
├── 概念：主动检测和修复信息不一致
├── 实现：定期 PING/PONG 交换 + CLUSTER NODES 全量同步
├── 触发：新节点加入、节点离线恢复、检测到冲突
└── 目的：确保最终一致性，防止 Gossip 遗漏
```

### 重定向机制

```
客户端访问流程：

Client ──get("key")──► NodeA (错误节点)
                          │
                          ▼ 计算槽位
                     CRC16("key") = 12345
                     12345 属于 NodeB
                          │
                          ▼
Client ◄────MOVED 12345 NodeB:6379────┘
     │
     ▼ 缓存槽位映射
Client ──get("key")──► NodeB (正确节点)
                          │
                          ▼
Client ◄────value─────────────────────┘

MOVED：永久重定向（槽位已迁移）
ASK：临时重定向（槽位正在迁移中）
```

### 故障转移

```
与哨兵模式的区别：
┌─────────────────┬─────────────────┐
│    哨兵模式      │   Cluster 模式   │
├─────────────────┼─────────────────┤
│ 专门的 Sentinel  │ 无中心，Master 互监控 │
│ 哨兵投票决策      │ Master 投票决策    │
│ 通知 Slave 提升  │ Slave 自己提升     │
└─────────────────┴─────────────────┘

Cluster 故障转移：
├── 故障检测：Master 之间互相 PING/PONG
├── 故障确认：多数 Master 标记为 FAIL
├── 选举：Slave 向所有 Master 请求投票
└── 提升：获得多数票后成为新 Master
```


## 关键关联

- [[Redis-哨兵模式]] - 关联原因：Cluster 和哨兵都提供高可用，但 Cluster 是去中心化设计
- [[Redis-主从复制]] - 关联原因：Cluster 内每个 Master-Slave 对使用主从复制同步数据
- [[数据分片]] - 关联原因：Cluster 使用哈希槽实现数据分片，水平扩展
- [[Gossip-协议]] - 关联原因：Cluster 使用 Gossip 协议进行去中心化的状态同步
- [[CAP-理论]] - 关联原因：Cluster 在 CAP 中选择 AP（可用性 + 分区容错）


## 我的误区与疑问

- ❌ 误区：以为 Cluster 是强一致的（实际是最终一致）
- ❓ 疑问：为什么不用 Raft 而用 Gossip？（去中心化、可扩展性优先）
- ❓ 疑问：槽位迁移期间数据一致性如何保证？（ASK 重定向 + 双写）
- ❓ 疑问：复制偏移量是共享的还是每个 Slave 独立的？（每个 Master-Slave 对独立维护，Gossip 传播用于选主）
- ❓ 疑问：Gossip 如何避免长期不一致？（Epoch 版本号 + 反熵机制确保收敛）


## 代码与实践

```bash
# 创建集群
redis-cli --cluster create \
  192.168.1.1:6379 192.168.1.2:6379 192.168.1.3:6379 \
  192.168.1.4:6379 192.168.1.5:6379 192.168.1.6:6379 \
  --cluster-replicas 1

# 查看集群状态
redis-cli -c -p 6379 CLUSTER INFO
redis-cli -c -p 6379 CLUSTER NODES

# 槽位迁移
redis-cli --cluster reshard 192.168.1.1:6379

# 添加新节点
redis-cli --cluster add-node 192.168.1.7:6379 192.168.1.1:6379
```
