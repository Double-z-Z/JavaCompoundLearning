# Redis 集群

## 一句话定义
Redis 集群是通过分片（Sharding）将数据分布在多个节点上的分布式存储方案，使用哈希槽（Hash Slot）实现数据路由。

## 核心理解

### 集群架构

```
┌─────────────────────────────────────────┐
│           Redis Cluster                 │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Master1 │ │ Master2 │ │ Master3 │   │
│  │ 0-5460  │ │5461-10922│ │10923-16383│ │
│  │ Slot    │ │  Slot   │ │  Slot   │   │
│  └────┬────┘ └────┬────┘ └────┬────┘   │
│       │           │           │         │
│  ┌────┴────┐ ┌────┴────┐ ┌────┴────┐   │
│  │ Slave1  │ │ Slave2  │ │ Slave3  │   │
│  │ (副本)  │ │ (副本)  │ │ (副本)  │   │
│  └─────────┘ └─────────┘ └─────────┘   │
└─────────────────────────────────────────┘
```

### 哈希槽（Hash Slot）

- **总数**：16384 个槽（0-16383）
- **计算**：`CRC16(key) % 16384`
- **分配**：每个 Master 负责一段连续的槽

### MOVED 重定向

当客户端请求的 key 不在当前节点时：

```
客户端 → 102 节点: GET key1
102 节点: MOVED 9189 10.0.0.103:6379
          ↑     ↑         ↑
        错误  槽位编号   正确节点

客户端 → 103 节点: GET key1
103 节点: 返回 value
```

使用 `redis-cli -c` 可以自动处理重定向。

### 故障转移

```
1. Master 102 故障
2. Slave 106 发现 Master 失联（通过 Gossip 协议）
3. Slave 106 发起选举
4. 其他 Master 投票
5. 106 成为新 Master，接管槽位 0-5460
6. 原 Master 恢复后成为 Slave
```

## 我的误区与疑问

- ❌ 曾以为集群需要代理层（如 Twemproxy）
  - ✅ Redis Cluster 是原生支持的，客户端直接连接任意节点
  
- ❌ 曾疑惑为什么槽是 16384 个而不是更多
  - ✅ 16384 = 2^14，Gossip 消息大小适中，且足够分散

## 代码与实践

```bash
# 创建集群
redis-cli --cluster create \
  10.0.0.102:6379 \
  10.0.0.103:6379 \
  10.0.0.104:6379 \
  10.0.0.105:6379 \
  10.0.0.106:6379 \
  10.0.0.107:6379 \
  --cluster-replicas 1

# 查看集群节点
redis-cli cluster nodes

# 查看集群信息
redis-cli cluster info
```

## 深入思考

💡 Redis 集群的设计权衡：
1. **无中心架构**：没有单点故障，但一致性需要 Gossip 协议维护
2. **槽位固定**：16384 个槽不可动态调整，但足够分散
3. **客户端路由**：MOVED 重定向增加了客户端复杂度，但减少了代理层开销
