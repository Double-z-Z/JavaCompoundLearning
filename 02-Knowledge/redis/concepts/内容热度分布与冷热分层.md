# 内容热度分布与冷热分层设计

> 基于真实 SNS 平台数据的热度分布规律，指导 Redis 计数器系统的冷热分层架构设计


## 一句话定义

内容热度服从**幂律分布**（Power Law），头部 10% 内容占据 80% 访问量，这种极度不均衡的分布是冷热分层架构的根本依据。


## 核心理解

### 1. 热度分布的数学规律

#### 静态分布：幂律分布（Power Law）

```
P(X=k) ∝ k^(-α)  where α ≈ 1.5-2.5
```

| 平台 | Gini 系数 | 头部 10% 占比 | 长尾特征 |
|------|----------|--------------|---------|
| YouTube | 0.82-0.89 | 80% 观看量 | 明显 |
| TikTok | ~0.93 | 90%+ 流量 | 极端 |
| Niconico | 0.85+ | 头部幂律+中尾对数正态 | 双模态 |

**关键洞察**：
- Gini 系数 > 0.8 表示极度不均衡
- 这种不均衡是**普遍规律**，与平台类型无关
- 为冷热分层提供了**量化依据**

#### 动态衰减：幂律衰减 vs 指数衰减

| 衰减类型 | 公式 | 特征 | 适用场景 |
|---------|------|------|---------|
| **幂律衰减** | Score = P / (T+2)^G | 初期快，后期慢 | 内容热度排序 |
| **指数衰减** | Score = P × e^(-λT) | 恒定半衰期 | 实时性要求高的场景 |

**Hacker News 公式**：
```
Score = (P-1) / (T+2)^1.8
- P: 投票数
- T: 发布时间（小时）
- G=1.8: 重力系数
```

**Reddit Hot 算法**：
```
每 12.5 小时失去相当于 10 倍 upvotes 的排名优势
使用对数投票 + 时间衰减
```

### 3. 热度衰减的工程实现

#### 方案 1：指数衰减计数器

```lua
-- 带半衰期的计数器
local key = KEYS[1]
local half_life = tonumber(ARGV[1])  -- 半衰期（秒）
local increment = tonumber(ARGV[2])

local current = redis.call('GET', key)
if not current then
    current = 0
else
    local parts = split(current, ':')
    local value = tonumber(parts[1])
    local timestamp = tonumber(parts[2])
    local elapsed = now() - timestamp
    -- 衰减公式: N = N0 * (1/2)^(t/t_half)
    local decay_factor = math.pow(0.5, elapsed / half_life)
    current = value * decay_factor
end

local new_value = current + increment
redis.call('SET', key, new_value .. ':' .. now())
return new_value
```

#### 方案 2：滑动窗口计数器

```
使用 Redis Hash，每个子区间一个 field：

key: counter:article:123:views
field          value
---------------|------
2024-05-02-14  | 150   (14:00-14:59)
2024-05-02-15  | 230   (15:00-15:59)
2024-05-02-16  | 180   (16:00-16:59)
...

最近 N 小时的总和 = 热数据评分
```

#### 方案 3：分层 TTL 策略

| 热度等级 | 判定条件 | TTL | 存储 |
|---------|---------|-----|------|
| 极热 | QPS > 100/min | 24h | Redis |
| 热 | QPS > 10/min | 7d | Redis |
| 温 | 有访问 | 30d | SSD |
| 冷 | 无访问 | 永久 | MySQL |

## 我的误区与疑问

### 误区 1：认为所有内容都需要缓存

**纠正**：基于幂律分布，只有 **Top 5-10%** 的内容值得放入 Redis

### 误区 2：使用固定阈值判断冷热

**纠正**：应该使用**动态阈值**，基于实时 QPS 调整

### 误区 3：忽视时间衰减

**纠正**：内容热度随时间**幂律衰减**，不是线性下降

### 热度计算服务

```java
@Service
public class HeatScoreCalculator {
    
    // Hacker News 风格的热度计算
    public double calculateHeatHN(long votes, long publishTimeHours) {
        double gravity = 1.8;
        return (votes - 1) / Math.pow(publishTimeHours + 2, gravity);
    }
    
    // 带半衰期的实时热度
    public double calculateHeatWithDecay(
            long currentValue, 
            long lastUpdateTime,
            long halfLifeSeconds) {
        long elapsed = System.currentTimeMillis() / 1000 - lastUpdateTime;
        double decayFactor = Math.pow(0.5, (double) elapsed / halfLifeSeconds);
        return currentValue * decayFactor;
    }
    
    // 滑动窗口热度
    public long calculateSlidingWindowHeat(String key, int hours) {
        long total = 0;
        long now = System.currentTimeMillis() / 3600000;
        for (int i = 0; i < hours; i++) {
            String field = String.valueOf(now - i);
            String value = redisTemplate.opsForHash().get(key, field);
            if (value != null) {
                total += Long.parseLong(value);
            }
        }
        return total;
    }
}
```

### 动态分层管理器

```java
@Component
public class DynamicTierManager {
    
    private static final double HOT_THRESHOLD_PERCENTILE = 0.95;  // Top 5%
    private static final double WARM_THRESHOLD_PERCENTILE = 0.80; // Top 20%
    
    @Scheduled(fixedRate = 60000) // 每分钟重新评估
    public void rebalance() {
        // 1. 收集所有内容的访问统计
        Map<String, AccessStats> allStats = collectStats();
        
        // 2. 按热度排序
        List<String> sortedKeys = allStats.entrySet().stream()
            .sorted((e1, e2) -> Double.compare(
                e2.getValue().getHeatScore(), 
                e1.getValue().getHeatScore()
            ))
            .map(Map.Entry::getKey)
            .collect(Collectors.toList());
        
        // 3. 确定分层边界
        int total = sortedKeys.size();
        int hotBoundary = (int) (total * (1 - HOT_THRESHOLD_PERCENTILE));
        int warmBoundary = (int) (total * (1 - WARM_THRESHOLD_PERCENTILE));
        
        // 4. 执行升降级
        for (int i = 0; i < total; i++) {
            String key = sortedKeys.get(i);
            if (i < hotBoundary) {
                promoteToHot(key);
            } else if (i < warmBoundary) {
                promoteToWarm(key);
            } else {
                demoteToCold(key);
            }
        }
    }
}
```


## 深入思考

### 为什么幂律分布普遍存在？

1. **富者愈富**（Preferential Attachment）：热门内容更容易被发现
2. **网络效应**：用户行为相互影响
3. **平台推荐算法**：放大头部效应

### 如何验证自己的业务是否符合幂律？

```python
import numpy as np
import matplotlib.pyplot as plt

# 1. 收集数据：内容的访问次数
views = [10000, 8000, 6000, ...]  # 按降序排列

# 2. 对数坐标检验
log_rank = np.log(range(1, len(views)+1))
log_views = np.log(views)

# 3. 线性回归
slope, intercept = np.polyfit(log_rank, log_views, 1)

# 4. 如果斜率接近 -1 到 -2，说明符合幂律
print(f"幂律指数 α = {-slope}")
```
