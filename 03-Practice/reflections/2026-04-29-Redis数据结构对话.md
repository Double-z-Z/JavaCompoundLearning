## 核心问答

### Q1: 为什么 SDS 比 StringBuilder 更"智能"？
**我的回答**：SDS 有预分配策略，小于1MB时2倍扩容，大于1MB时只+1MB，还有惰性释放。

**AI补充**：
- StringBuilder 固定2倍扩容，大数据时内存暴涨
- SDS 的增长抑制策略控制内存占用
- 预分配缓存减少频繁申请/释放

**顿悟**：空间换时间的思想，类似 ThreadLocal 和对象池

---

### Q2: 跳表的随机层数为什么能保证 O(log n)？
**我的回答**：每个节点以概率 p=0.25 升级，期望层数 1.33，最大层数 O(log n)。

**AI引导**：严格证明需要解 n·p^(k-1) ≈ 1

**推导过程**：
```
p^(k-1) = 1/n
(k-1)·ln(p) = -ln(n)
k-1 = ln(n) / ln(1/p)
k = log_{1/p}(n) + 1 = O(log n)
```

**顿悟**：概率分布的期望分析，几何分布的妙用

---

### Q3: 渐进式 rehash 和 Java HashMap 扩容的区别？
**我的回答**：
- Redis：分步迁移，每次操作顺带迁移一个桶，查询需查两个表
- Java：一次性扩容，阻塞式，需要重新计算所有哈希

**AI补充**：
- Redis 单线程模型适合渐进式
- Java 多线程环境需要快速完成避免长时间锁
- 渐进式 rehash 期间，新数据只写入新表，保证不会重复迁移

**顿悟**：设计选择取决于使用场景（单线程 vs 多线程）

## 思维模型升级

### 之前的理解
```
Redis = 键值存储
  ├─ String 存字符串
  ├─ Hash 存对象
  ├─ List 存列表
  ├─ Set 存集合
  └─ ZSet 存有序集合
```

### 现在的理解
```
Redis = 多编码自适应存储系统
  ├─ 运行时根据数据特征选择最优编码
  ├─ 小数据：紧凑编码（ziplist/intset）
  ├─ 大数据：高效编码（hashtable/skiplist）
  ├─ 透明切换，用户无感知
  └─ 空间与性能的动态权衡

跳表 = 概率平衡的艺术
  ├─ 用随机代替复杂的平衡逻辑
  ├─ 期望性能 O(log n)
  ├─ 实现简单，代码量少
  └─ 天然支持范围查询和排名
```

### 应用价值
- 设计存储系统时考虑多种编码自适应
- 用概率算法简化复杂数据结构的实现
- 单线程模型下渐进式操作避免阻塞
